## Branch Protection Interpretation

The 2026-06 post-transfer signoff correctly observed that GitHub reported
`protected: false` for `main` at that time. That remains useful historical
evidence, but it should not block the visibility switch when branch protection
is unavailable solely because the repository is private on the current plan.

Updated release policy:

- if GitHub allows private-repository branch protection, enable or confirm
  `main` protection before public opening;
- if GitHub only allows branch protection after the repository becomes public,
  record the intended `main` protection policy before opening, switch
  visibility only after the other gates pass, then enable protection
  immediately after the public switch and before inviting external
  contributors or announcing the repository broadly.

This preserves the safety goal without making an impossible private-repository
setting a false blocker.
