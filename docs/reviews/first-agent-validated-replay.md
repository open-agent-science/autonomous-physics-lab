## Boundary

This document records the **first live use of Gate B** in the multi-tier
result-promotion protocol introduced by `TASK-0406`. It is the first time an
existing canonical `AGENT_PUBLISHED` result is independently replayed and
upgraded to `AGENT_VALIDATED` through the agent-to-agent validation layer,
without any maintainer endorsement and without any claim or knowledge
promotion.

It does **not** modify the underlying experimental evidence, scientific
metrics, verdict, train/test windows, scores, verification checks, or
limitations recorded in `RESULT-0016`. It does **not** touch
`CLAIM-0009-anharmonic-oscillator-period.md` and does **not** trigger any
claim-status transition. It does **not** pin the result to
`results/golden-results.yaml`.

## Cross-tool disclosure

The `agent-validated-result-checklist` states:

> Prefer cross-tool validation (Claude Code validates a Codex result, or
> Codex validates a Claude Code result). Same-tool replay is acceptable only
> when clearly disclosed.

This replay is **same-tool**: a Claude Code session validated a Claude Code
result, although the replaying session is distinct from the publishing
session (different timestamps, different branch, different conversation).
Per the checklist, this is acceptable when disclosed; the disclosure is
recorded here and in the `agent_proposal_evaluation.followup_for_maintainer`
field of `RESULT-0016`. Future Gate B replays of subsequent AGENT_PUBLISHED
results should prefer cross-tool validation (Codex replaying a Claude Code
result, or vice versa) once a Codex-side replay runner is exercised.

## Warning recorded

The Gate B runner emitted one advisory warning during replay:

- `WARNING original-publisher-unknown`: Original publisher metadata
  (`replayed_by`-equivalent identity for the AGENT_PUBLISHED author) is not
  stored on `RESULT-0016`. The runner records replay identity but cannot
  mechanically compare originating-agent vs replaying-agent.

This is a protocol-level observation, not a Gate B failure. The historical
TASK-0412 PR (#581) records the publishing identity in PR metadata, so the
human audit trail is complete; only mechanical agent-identity comparison
inside the RESULT artifact is unavailable. A future small protocol
improvement could add an `agent_proposal_evaluation.proposed_by` block to
the Gate A template to mirror `validation_record.replayed_by`. This is
flagged here as a follow-up; **no schema change is included in this PR**.

## Limitations

- Same-tool replay (Claude Code → Claude Code) is permitted-with-disclosure
  by the checklist, but is intentionally weaker than cross-tool replay.
  Future replays should prefer cross-tool.
- Original publisher metadata is missing from RESULT-0016 itself; the
  human audit trail relies on TASK-0412 PR #581 rather than on
  machine-readable identity comparison inside the result file.
- The Gate B runner records `replay_output_dir` as a machine-temporary
  path; this PR strips that field before writing the canonical
  `validation_record` to avoid leaking local filesystem state. The full
  runner output is preserved in this review document instead.
- Tolerance was the runner default (`1.0e-09`); no per-result tolerance
  override was requested or recorded. For workflows whose deterministic
  output drifts at the floating-point ULP scale, a higher tolerance may
  be appropriate; this was not necessary here (`max_abs_delta = 0.0`).
- Gate B does not test causal model interpretation. A bit-for-bit
  reproduction confirms determinism and input pinning, not physical
  validity. Physical interpretation remains under maintainer review for
  `MAINTAINER_REVIEWED` and external replication for
  `EXTERNAL_REPLICATED`.
