## Helper Path-Layout Fix Required To Run Gate B

Before this task, the Gate B helper could not even locate the replayed result
for this workflow. The helper assumed every workflow writes
`<output-dir>/EXP-XXXX/RUN-XXXX/result.yaml` (nested). The dimensional analysis
validator intentionally writes `result.yaml` flat in `<output-dir>` (see commit
`abb520b8 fix(task-0064): emit binomial uncertainty and stop double-nesting
result dir`). The helper therefore returned `BLOCKED: replay-result-missing`
and masked the real reproducibility state.

This task fixes `_replayed_result_path` in
`physics_lab/registry/agent_replay_validation.py` to accept both the nested and
the flat layout, with a regression test
(`tests/test_agent_replay_validation.py::test_gate_b_finds_flat_layout_result`).
The fix is layout-only; it does not relax any numeric, verdict, or provenance
comparison. Running the corrected helper is what surfaced the contestation
documented below.

### Contesting issue

- `metric-missing`: the replay does not regenerate
  `verification.checks[4].metrics.disagreement_count`.

A fresh `python -m physics_lab.cli run examples/dimensional_analysis_live_74.yaml`
produces **4** verification checks:

1. `challenge_set_loaded`
2. `challenge_set_declared_total_matches_items`
3. `inconclusive_items_within_mvp_tolerance`
4. `agreement_fraction`

The committed `RESULT-0020` artifact contains **7** verification checks. The
three additional checks are not produced by the recorded command:

5. `zero_disagreement_ledger`
6. `frozen_input_checksum`
7. `protected_result_not_rewritten`

A repository search finds **zero** occurrences of `zero_disagreement_ledger` or
`frozen_input_checksum` in any code path (`physics_lab/`, `scripts/`,
`examples/`), and `protected_result_not_rewritten` is emitted only by the
unrelated `physics_lab/workflows/textbook_exact_reference.py`. These three
checks were therefore hand-authored during the `TASK-0750` Gate A packaging and
are not deterministically reproducible from `RESULT-0020`'s own recorded
`command`.

## Interpretation

The **scientific core** of `RESULT-0020` reproduces cleanly: the deterministic
74-item dimensional-consistency replay agrees with the curated benchmark on
74/74 items, the `VALID` verdict holds, and every reproduced numeric metric
matches with zero drift. The dimensional-consistency evidence itself is sound.

What Gate B contests is **provenance/packaging integrity**, not the science:
part of the published `verification` block (the publication-gate checks) is not
regenerated by the recorded command, so the artifact as committed is not fully
reproducible end-to-end. Gate B is designed to catch exactly this, so the
correct outcome is `CONTESTED_RESULT`, not an `AGENT_VALIDATED` upgrade.

As required by `result-promotion-protocol.md`, dimension-only agreement remains
formula-quality evidence, not proof of numerical correctness, empirical
validity, or physical truth — independent of the packaging issue.

## Recommended Maintainer Follow-Up

To convert `RESULT-0020` into Gate-B-passable memory, a separate maintainer-owned
follow-up should choose one of:

1. **Repackage reproducibly** — make the Gate A packaging step emit the
   `zero_disagreement_ledger`, `frozen_input_checksum`, and
   `protected_result_not_rewritten` checks deterministically (a packaging helper
   or workflow flag), then re-run Gate B; or
2. **Separate packaging annotations from run output** — store hand-authored
   publication-gate checks in a clearly labelled packaging/Gate-A section that
   the Gate B comparator does not treat as reproducible run metrics, then re-run
   Gate B against the reproducible core only.

Either path is out of scope for this replay-validation task and must not silently
rewrite the committed `RESULT-0020` artifact.
