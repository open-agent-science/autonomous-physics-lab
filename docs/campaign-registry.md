# Campaign Registry

## Purpose

`campaign_profiles/_catalog.yaml` is the lightweight machine-readable portfolio
map for APL campaign lanes. It exists so mission control, snapshots, curators,
and parallel agents can inspect campaign state without reconstructing it from
narrative pages every time.

The index is generated by `scripts/generate_campaign_catalog.py` from
`campaign_profiles/*.yaml` `portfolio` blocks. Do not edit the generated index
by hand. Edit the campaign profile, then regenerate the index.

The registry is intentionally small. It is not a database, dashboard, task
queue, or replacement for campaign profiles.

## Relationship To Existing Files

`missions/current.yaml` remains the live recommendation surface. It decides what
new agents should see first and which tasks are currently preferred.

`campaign_profiles/` remains the autonomy contract surface. A profile says what
bounded work an agent may do in a campaign and which claims or artifacts are
forbidden.

`docs/campaigns/` remains the human-readable campaign map. Those pages explain
the scientific context, evidence trail, and limitations.

`campaign_profiles/_catalog.yaml` sits between them as a generated view:

```text
missions/current.yaml        live recommendations and priorities
campaign_profiles/*.yaml     autonomy contracts plus portfolio source fields
campaign_profiles/_catalog.yaml generated machine-readable portfolio state
docs/campaigns/              human-readable campaign narratives
```

When these files disagree, use `missions/current.yaml` for current task
recommendations and the campaign profile for allowed/forbidden agent work. The
profile should then be updated and the index regenerated to match the
reviewed state.

The leading underscore is intentional: `_catalog.yaml` is generated aggregate
state, not an editable campaign profile. The generator ignores `_*.yaml` files
when discovering profiles.

## Required Fields

Each generated campaign entry records:

- `id` and `title`;
- `status`;
- `domain`;
- `surface_type`;
- `lifecycle_stage`;
- `activity_status`;
- `curator`;
- `current_stage`;
- `agent_capacity`;
- `best_next_actions`;
- `required_gates`;
- `allowed_work`;
- `forbidden_work`;
- `current_results`;
- `open_questions`;
- `curator_review`.

`domain` is the science domain. `curator.primary_pool` is the operational
ownership shard for focused Scientific Campaign Director sessions. These are
different dimensions; do not use domain values as curator-session groups.
For pool vocabulary and transfer rules, see
[Campaign Curator Pools](./campaign-curator-pools.md).

The schema lives at
`physics_lab/schemas/campaign_catalog.schema.json`. Repository validation loads
`campaign_profiles/_catalog.yaml` through `physics_lab.registry.campaigns` and
rejects missing required fields. The loader also rejects duplicate campaign ids.

The generator copies `allowed_work` from each profile's
`allowed_hypothesis_families` plus `allowed_experiment_families`, and copies
`forbidden_work` from `forbidden_claims`. This keeps the catalog from
duplicating safety rules already owned by the campaign profile.

## Status Vocabulary

Use these status values:

- `flagship_validation` - the strongest current scientific validation surface;
- `active_secondary` - active benchmark surface with visible public value;
- `source_readiness` - source/data readiness before benchmark metrics;
- `pinned_dataset` - source rows exist but benchmark gates are not complete;
- `scaffold` - campaign exists but first source/baseline slice is still being
  defined;
- `quality_floor` - mature benchmark used for replay, examples, or regression
  discipline;
- `guardrail` - constrained safety or falsification lane;
- `watchlist` - future direction, not executable without a task.

## Agent Capacity

`agent_capacity.recommended_parallel_agents` is advisory. It should not be used
as a scheduler limit. It tells a mission or curator agent how much parallel work
the campaign can absorb without duplicate artifacts or hidden conflicts.

Good capacity notes name the safe split:

- separate source ingestion vs loader vs holdout tasks;
- replay vs evidence-card packaging;
- negative-result packaging vs fresh diagnostic lanes.

Bad capacity notes say only "more agents welcome" without naming disjoint
artifact surfaces.

## Curator Pools

Use `curator.primary_pool` to run focused curator sessions without changing the
Scientific Campaign Director prompt:

```bash
python3 scripts/apl_campaign_curator.py --pool source_data_benchmark
python3 scripts/apl_campaign_curator.py --domain astrophysics
python3 scripts/apl_campaign_curator.py --stage source_readiness --active-only
```

`--pool` matches primary ownership only. `curator.secondary_pools` provides
handoff context but does not automatically include a campaign in another
session. Pool transfers require a maintainer or Scientific Campaign
Director-approved PR; scripts may filter or warn but must not mutate
assignment.

## Update Rules

Update the catalog when:

- a campaign changes maturity stage;
- mission control starts recommending a new campaign lane;
- a source gate is opened, closed, or superseded;
- a public-safe result card or scorecard changes the evidence surface;
- a curator review changes allowed next work.

Do not update the catalog for every task status transition. The task registry
and generated task views already handle that level of detail.

To update it:

```bash
python3 scripts/generate_campaign_catalog.py --write
```

To check it without writing:

```bash
python3 scripts/generate_campaign_catalog.py --check
```

## Validation

Run:

```bash
python3 -m pytest tests/test_campaign_registry.py
python3 scripts/generate_campaign_catalog.py --check
python3 -m physics_lab.cli validate-repo . --strict --fail-on-warnings
```

Full task handoff should still use the task's validation commands.
